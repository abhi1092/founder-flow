# FounderFlow — Multi-Agent Startup Validation Assistant

## Vision
FounderFlow treats a founder's startup idea as a research hypothesis and validates it through an iterative multi-agent research loop — idea validation, competitor analysis, and customer discovery agents fan out in parallel, then an Evidence Integrator acts as a research advisor that identifies gaps and sends specific agents back for deeper investigation until evidence converges. The system produces a polished, self-contained HTML Startup Validation Research Brief with contradictions exposed, assumptions surfaced, and a concrete 7-day action plan.

## Core Features

- **Iterative Multi-Agent Research Loop**
  - **What:** The founder submits a startup idea in plain English. FounderFlow executes an iterative research loop that begins with a broad, shallow pass (Round 1) and progressively deepens through directed re-investigation rounds until evidence converges or a configurable iteration cap is reached. **Round 1 (Broad, Shallow):** Three specialist agents run simultaneously — an Idea Validator (evaluates problem severity, target customer clarity, value proposition strength, risk factors, and core assumptions), a Competitor Agent (maps direct competitors, indirect competitors, substitutes, current workarounds, pricing patterns, and positioning gaps), and a Customer Discovery Agent (designs interview targets, discovery questions, demand signals, weak-signal warnings, and a first validation experiment). A barrier waits for all three. **Round 2+ (Narrow, Deep):** The Evidence Integrator reviews all outputs and produces a `ResearchDirective` — a structured list of specific follow-up questions routed to specific agents. Only agents with identified gaps re-run with refined, targeted prompts; agents with sufficient evidence sit idle and are not re-invoked. Each re-invoked agent receives its own prior output plus the Integrator's directive, so it builds on prior work rather than starting fresh. **Convergence:** The loop terminates when the Integrator declares `evidence_sufficient` (no unresolved contradictions, key assumptions identified), the user-configured `--max-rounds` cap is reached (default: 3), or the token budget is exhausted. The founder sees a Rich progress display showing the current round number, which agents are active vs. idle, live status updates, and per-round cost accumulation.
  - **How:** The orchestrator in `pipeline.py` implements the research loop as an async state machine. Round 1 uses `asyncio.gather()` to fan out all three agents concurrently, each spawned as a Claude Code subprocess via `claude -p` with a focused system prompt loaded from `founderflow/agents/prompts/<role>.md`. Agent outputs are written to `.founderflow/runs/<id>/rounds/<round_num>/<role>.json` as Pydantic v2 models with `ConfigDict(strict=True, extra="forbid")`. After each round's barrier, the Evidence Integrator runs and produces a `ResearchDirective` (written to `.founderflow/runs/<id>/rounds/<round_num>/directive.json`). If the directive's verdict is `needs_deeper`, the orchestrator inspects `directive.directives` — a list of `{agent_role, follow_up_question, priority}` tuples — and spawns only the listed agents for the next round, injecting their prior-round output and the follow-up question into their prompt context. A `RoundResult` model captures each round's agent outputs, directive, costs, and timing. The loop controller in `pipeline.py` checks three termination conditions after each Integrator pass: (1) `directive.verdict == "evidence_sufficient"`, (2) `round_num >= max_rounds`, (3) cumulative token cost exceeds budget. Agent prompts are resolved via two-tier lookup: project override at `.founderflow/agents/<role>.md` falls back to built-in default at `founderflow/agents/prompts/<role>.md`.
  - **Why:** Research identifies fan-out/fan-in as the optimal pattern when all specialists must contribute — wall-clock time equals the slowest agent, not the sum. But a single-pass fan-out cannot detect when evidence is thin or contradictory in a specific dimension. The iterative loop mirrors the academic peer review process: Round 1 is the initial literature review, the Integrator is the peer reviewer identifying gaps, Round 2+ addresses reviewer comments, and convergence means the paper is accepted. This architecture avoids the waste of re-running agents that already have sufficient evidence (only gap-bearing agents re-run) while ensuring the Integrator can drive arbitrarily deep investigation on specific concerns. The Factory reference implementation proves that Claude Code subprocesses with file-based communication eliminate the need for message buses or framework dependencies while maintaining clean agent isolation. LangGraph's conditional routing could model this loop, but the fixed three-agent topology with a simple directive-driven re-invocation pattern doesn't justify the framework dependency — a Python `while` loop with conditional `asyncio.gather()` is sufficient and more transparent.

- **Evidence Integrator / Research Advisor**
  - **What:** The Evidence Integrator acts as a research advisor that reviews all agent outputs after each round and makes one of two decisions: declare evidence sufficient and proceed to synthesis, or produce a `ResearchDirective` that sends specific agents back for deeper investigation. After Round 1, the Integrator performs cross-agent critique: identifying contradictions (e.g., Idea Validator says the problem is painful but Competitor Agent finds existing tools already solve it), highlighting unresolved assumptions, and spotting thin evidence. Rather than simply flagging these issues in a report, it formulates specific follow-up questions routed to the agent best positioned to resolve each gap. After Round 2+, the Integrator evaluates whether the new evidence resolved the gaps from the prior directive, logs which contradictions were clarified, identifies any remaining gaps, and decides whether another round is warranted. On convergence (or max rounds), it produces the final `ValidationThesis` with a confidence score and go/deeper/pivot/kill verdict.
  - **How:** The Integrator agent receives a composable prompt assembled from: a base research advisor prompt, all agent outputs from the current round (loaded from `.founderflow/runs/<id>/rounds/<round_num>/<role>.json`), the prior round's directive (if any), and a structured output schema. After each round, it outputs a `ResearchDirective` Pydantic model containing: `verdict` (enum: `needs_deeper` | `evidence_sufficient`), `directives` (list of `FollowUpDirective` — each specifying `agent_role`, `follow_up_question`, and `priority: high | medium`), `resolved_contradictions` (what was clarified this round), `remaining_gaps` (what still needs work), and `round_confidence` (0-100 score for the current evidence quality). Only on the terminal round (convergence or cap) does the Integrator additionally produce a `ValidationThesis` containing: thesis statement, final confidence score 0-100, verdict enum, full contradictions list, unresolved assumptions list, synthesized risk assessment, and a summary of the research journey across rounds. The Integrator writes its per-round output to `.founderflow/runs/<id>/rounds/<round_num>/directive.json` and the final synthesis to `.founderflow/runs/<id>/synthesis.json`. Context is managed via tiered history — the current round gets full agent outputs, prior rounds get the directive summaries and resolved contradictions only, keeping the Integrator's context window focused.
  - **Why:** The Factory's Pattern 12 (file-based agent communication) proves that agents reading structured JSON files produces more reliable handoffs than direct inter-agent messaging. The research advisor role goes beyond a simple critic — it actively steers the research process, mirroring how a thesis advisor identifies gaps and assigns specific follow-up work rather than just noting problems. This is the key architectural distinction from single-pass validation tools like IdeaProof (120-second single-pass) and even multi-agent systems that run once without feedback loops. The `ResearchDirective` model makes the Integrator's reasoning explicit and inspectable — the founder can read the directive JSON to understand exactly why certain agents were asked to dig deeper and on what questions.

- **Three-Tier Brief Delivery (HTML-Primary)**
  - **What:** The system produces the Startup Validation Research Brief in three tiers, each serving a different purpose. **Tier 1 — Terminal (immediate):** Rich-formatted output displayed during and after the pipeline run. Color-coded verdict panel (green background = go, yellow = dig deeper, red = pivot/kill), risk indicator bars, structured section panels with clear visual hierarchy, and a research loop summary showing how many rounds were executed, which agents were active in each round, and what gaps were resolved. **Tier 2 — Self-contained HTML (primary deliverable):** A single `.html` file with all CSS embedded inline — no external dependencies, no CDN links, no external fonts requiring internet access. The file opens in any browser by double-clicking, can be emailed to a co-founder, mentor, or advisor, and looks professional and polished: clean typography, color-coded risk levels (green/yellow/red badges), visual section hierarchy with cards and dividers, a styled 7-day action plan table, a research loop timeline showing the progression of evidence across rounds, and a cost summary footer with per-round breakdown. **Tier 3 — JSON (machine-parseable):** The structured `brief.json` containing all data for programmatic access — importing into project management tools, feeding into future Builder agents, or building dashboards.
  - **How:** The `BriefRenderer` class in `rendering.py` takes the `ValidationThesis`, all agent outputs from the final round, and the full list of `RoundResult` objects as input. It produces three outputs: (1) Rich console output via `rich.panel.Panel`, `rich.table.Table`, and `rich.markdown.Markdown` renderers with a color theme mapped to the verdict enum, including a "Research Journey" panel showing round-by-round progression, (2) an HTML file rendered from a Jinja2 template at `founderflow/templates/brief.html.j2` with all styles in an inline `<style>` block — the template uses a clean sans-serif font stack (`-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif`), a constrained content width (max 800px centered), and a color palette derived from the verdict (green `#22c55e` / yellow `#eab308` / red `#ef4444` as accent colors against a white/light-gray background), with a collapsible "Research Loop Detail" section showing per-round directives and convergence path, (3) a structured JSON file. All three are written to `.founderflow/runs/<id>/` as `brief.html`, `brief.json`. The HTML also embeds a cost summary (total tokens, per-agent and per-round breakdown, dollar cost) and a generation timestamp. After writing the HTML, the CLI auto-opens it in the default browser via `webbrowser.open()`.
  - **Why:** Research on similar platforms (FounderSpace, IdeaProof) shows they produce flat reports without structured action plans or shareable artifacts. A self-contained HTML file is the ideal shareable format: it requires zero infrastructure (no server, no app install), renders identically on every device, and looks polished enough to forward to an advisor or investor. The three-tier approach serves all audiences — terminal for the founder's immediate workflow, HTML for sharing and revisiting, JSON for programmatic downstream use. The research loop timeline in the HTML brief is a differentiating artifact — it shows the founder (and their advisors) that the validation wasn't a single shallow pass but an iterative deepening process, building confidence in the rigor of the analysis.

- **Append-Only Event Log and Cost Tracking**
  - **What:** Every system action — agent spawns, completions, failures, timeouts, stage transitions, round lifecycle events, and costs — is recorded in an append-only JSONL event log. The user can inspect any run's full history, see per-agent and per-round token usage and dollar costs, and the system can recover from crashes by replaying the event log to reconstruct state. Round-level events capture the research loop's progression: which agents were active, what the Integrator directed, and the cost of each round — enabling the founder to see diminishing returns and understand whether additional rounds added meaningful evidence.
  - **How:** The `events.py` module implements `emit_event(run_path, event_type, *, agent=None, round_num=None, data=None)` writing to `.founderflow/runs/<id>/events.jsonl` with event types: `agent.started`, `agent.completed`, `agent.failed`, `agent.timeout`, `round.started` (with `active_agents` list), `round.completed` (with round cost and duration), `round.convergence_check` (with directive verdict and remaining gaps count), `pipeline.started`, `pipeline.completed`, `gate.passed`, `gate.failed`, `cost.accumulated`. Every agent invocation returns an `AgentUsage` Pydantic model tracking `input_tokens`, `output_tokens`, `cache_read_tokens`, `cache_creation_tokens`, `total_cost_usd`, `duration_ms`, `num_turns`, and `model`. A `sum_agent_costs(run_path)` function aggregates costs across all agents in a run, and a `sum_round_costs(run_path, round_num)` function aggregates costs for a specific round. A companion `runs.tsv` file at `.founderflow/runs.tsv` appends one row per completed run for cross-run analytics, now including `num_rounds` and `converged` columns.
  - **Why:** The Factory's Pattern 2 (append-only event log + TSV results) provides crash recovery, cost tracking, and audit trail with zero infrastructure — no database required. Per-round cost tracking is especially important for the iterative loop: founders can see that Round 1 cost $0.45, Round 2 cost $0.22 (only two agents re-ran), and Round 3 cost $0.08 (one agent), making diminishing returns visible and justifying the `--max-rounds` default.

- **State Machine for Run Lifecycle**
  - **What:** Each validation run progresses through an explicit state machine: `SUBMITTED → VALIDATING → DEEPENING → SYNTHESIZING → RENDERING → COMPLETE` (happy path) or error states at any point. `VALIDATING` covers Round 1 (the initial broad fan-out of all three agents plus the first Integrator pass). `DEEPENING` covers Rounds 2+ (directed re-investigation rounds where specific agents are re-invoked based on the Integrator's `ResearchDirective`). The transition from `VALIDATING` to `DEEPENING` occurs when the Integrator's Round 1 directive has verdict `needs_deeper`. The transition from `DEEPENING` to `SYNTHESIZING` occurs when the Integrator declares `evidence_sufficient` or the max-rounds cap is reached. The `RENDERING` state covers brief generation across all three tiers. The CLI displays current state and round number, and runs can be resumed from the last successful state after a crash.
  - **How:** A `RunState` enum in `state.py` defines all valid states: `SUBMITTED`, `VALIDATING`, `DEEPENING`, `SYNTHESIZING`, `RENDERING`, `COMPLETE`, `FAILED`, `ERRORED`. State is persisted to `.founderflow/runs/<id>/state.json` with the current state, current round number, timestamps for each transition, agent completion flags per round, and the latest directive verdict. The `detect_state(run_path)` function reads filesystem markers (presence of round directories, agent output files, directive files, event log entries) to reconstruct state — identical to the Factory's Pattern 5 approach of detection over assertion. `filelock.FileLock` protects concurrent writes to state files, following Factory Pattern 8. On crash recovery, the system detects which round was in progress and which agents within that round had completed, allowing it to resume from the exact point of failure rather than restarting the entire round.
  - **Why:** The Factory's state machine pattern (Pattern 5) combined with file-locked persistence (Pattern 8) enables crash-resilient long-running workflows without a database. The `DEEPENING` state is critical for the iterative loop — it distinguishes the initial broad investigation from the directed re-investigation rounds, giving the CLI and event log a clear signal of the pipeline's phase. State detection from filesystem markers means the system self-heals — if a crash corrupts `state.json`, the detection logic rebuilds state from the round directories, output files, and event log entries. The per-round crash recovery is essential because later rounds are cheaper (fewer agents running), so re-running an entire multi-round pipeline on crash would waste the expensive Round 1 work.

- **Pluggable Agent Runner Protocol**
  - **What:** Agent execution is abstracted behind a `Runner` protocol, allowing the system to swap between LLM backends (Claude, OpenAI, local models) without changing orchestration logic. The default runner uses Claude Code subprocesses (`claude -p`), but alternative runners can be configured per-agent or globally via config.
  - **How:** `agents/protocol.py` defines a `Runner` protocol class with `async headless(prompt, task, cwd, *, timeout, model) -> tuple[str, int, AgentUsage | None]` matching the Factory's Pattern 4 exactly. `runners/claude.py` implements `ClaudeRunner` as the primary backend, spawning `claude -p` subprocesses with structured prompts and parsing stdout for the agent's response. Configuration in `.founderflow/config.toml` (read via Python 3.11+ `tomllib`) specifies which runner to use per agent role, enabling cost optimization: e.g., route the three parallel agents through a cheaper model while keeping the Evidence Integrator on a capable model.
  - **Why:** The Factory's Runner Protocol (Pattern 4) with three interchangeable implementations (`ClaudeRunner`, `BobRunner`, `CodexRunner`) proves this abstraction works in production. Research recommends using cheap models for filtering stages and capable models for complex reasoning — the Runner protocol makes this trivial to configure. For the iterative loop, this is especially valuable: Round 1 agents doing broad sweeps can use a cheaper model, while the Integrator (which must perform nuanced cross-agent critique and produce structured directives) warrants a more capable model.

- **Typer + Rich CLI with Live Progress and Auto-Open**
  - **What:** The primary interface is a polished CLI built with Typer and Rich. The main command `founderflow validate "my startup idea"` runs the full pipeline, displays a live progress panel showing the current round number, all active agents' status (pending/running/complete/idle), the Evidence Integrator's progress, token counts, elapsed time, and per-round cost accumulation. Agents that the Integrator has deemed sufficient in a prior round show as "idle" with a checkmark rather than "pending" — making it clear they're sitting out by design, not due to error. On completion, the terminal displays the Tier 1 rich-formatted brief and automatically opens the Tier 2 HTML brief in the user's default browser. Additional commands: `founderflow runs` (list past runs with verdict badges and round counts), `founderflow show <id>` (re-open a past brief's HTML in the browser), `founderflow show <id> --terminal` (redisplay the terminal-formatted brief), `founderflow show <id> --json` (print raw JSON), `founderflow costs` (aggregate cost report across runs with per-round breakdown).
  - **How:** `cli.py` is a thin Typer wrapper — each command is a 5-10 line function that delegates to library modules (`pipeline.py`, `store.py`, `rendering.py`). Rich `Live` display with a `Table` layout shows real-time agent progress, updated per-round. The round indicator shows "Round 1/3 (broad)" or "Round 2/3 (directed: idea_validator, competitor_analyst)" to make the loop state visible. The verdict is displayed as a large color-coded Rich `Panel` (green/yellow/red background). After the brief files are written, `webbrowser.open(brief_html_path)` auto-opens the HTML in the default browser. Key CLI flags: `--max-rounds N` (default 3, controls the iteration cap), `--no-open` (suppresses auto-open for headless/CI environments). The CLI follows the Factory's Pattern 10: all business logic lives in library modules, CLI handlers only parse arguments and format output. Structured logging via `structlog` (Pattern 9) outputs JSON logs with automatic context binding for every agent lifecycle event.
  - **Why:** The Factory's 4,120-line monolithic CLI is identified in research as an anti-pattern. Typer provides the same CLI power with dramatically less boilerplate, and the early decision to keep the CLI thin prevents the same bloat. Rich provides the visual polish that makes the iterative research loop legible to the user — seeing three agents work simultaneously in Round 1, then a targeted subset re-investigate in Round 2, with real-time cost accumulation, is the key differentiator over single-pass tools like IdeaProof. The round-aware progress display transforms what could be an opaque multi-minute process into a transparent research journey the founder can follow in real time.

- **Two-Tier Prompt Resolution with Composable Sections**
  - **What:** Agent prompts are resolved in priority order — project-level overrides take precedence over built-in defaults, allowing users to customize agent behavior (e.g., focus on B2B SaaS validation, or healthcare-specific competitor analysis) without modifying source code. Prompts are composed from reusable sections rather than monolithic files. For Round 2+ re-invocations, the agent's prompt is dynamically augmented with its prior-round output and the Integrator's follow-up question, so it builds on prior work rather than starting fresh.
  - **How:** `agents/runner.py` implements prompt resolution: first checks `.founderflow/agents/<role>.md`, then falls back to `founderflow/agents/prompts/<role>.md`. Prompts use a section-include syntax (`{{section:output_format}}`) to compose from shared sections in `founderflow/agents/prompts/sections/` — output format instructions, grounding rules, structured output schemas. For re-invocation rounds, the runner prepends a "Prior Work" section containing the agent's last output and appends the Integrator's specific follow-up question from the `ResearchDirective`, creating a focused, additive prompt that avoids repeating broad investigation.
  - **Why:** Research identifies composable prompts as a fix for the Factory's monolithic prompt anti-pattern (500+ line files that are hard to maintain). The two-tier resolution from Pattern 3 is preserved for user customization, but the section composition system keeps individual prompt files focused and reusable across agent roles. The additive prompt construction for re-invocation rounds is critical to the loop's efficiency — without it, agents would repeat their Round 1 work verbatim, wasting tokens and wall-clock time.

- **File-Based Agent Communication via Structured JSON**
  - **What:** Agents never communicate directly. The orchestrator writes task context into each agent's prompt and agents write their structured output to designated JSON files. The Evidence Integrator reads the agent output files from the current round (and directive files from prior rounds) to perform its analysis. The round-based directory structure creates a clear, inspectable audit trail — users can read any agent's raw output from any round to understand how the evidence evolved across the research loop.
  - **How:** Each run creates a directory at `.founderflow/runs/<id>/` containing round-specific subdirectories and run-level synthesis files. The structure is:
    ```
    .founderflow/runs/<id>/
    ├── input.json                          # Original idea + config (max_rounds, etc.)
    ├── state.json                          # Current state, round number, timestamps
    ├── events.jsonl                        # Full event log
    ├── rounds/
    │   ├── 1/
    │   │   ├── idea_validation.json        # Idea Validator output
    │   │   ├── competitor_analysis.json    # Competitor Agent output
    │   │   ├── customer_discovery.json     # Customer Discovery Agent output
    │   │   ├── directive.json             # Integrator's ResearchDirective
    │   │   └── usage.json                 # Per-round cost aggregation
    │   ├── 2/
    │   │   ├── idea_validation.json        # Only if re-invoked by directive
    │   │   ├── directive.json
    │   │   └── usage.json
    │   └── .../
    ├── synthesis.json                      # Final ValidationThesis (after convergence)
    ├── brief.html                          # Self-contained HTML brief
    ├── brief.json                          # Machine-parseable JSON brief
    └── usage.json                          # Aggregate cost across all rounds
    ```
    All inter-agent data structures are strict Pydantic v2 models defined in `models.py`. Using `extra="forbid"` catches schema drift immediately. Round directories contain only the agents that actually ran in that round — the absence of a file (e.g., no `competitor_analysis.json` in Round 2) is a clear signal that the Integrator deemed that agent's evidence sufficient.
  - **Why:** The Factory's Pattern 12 (agent communication via state files) and Pattern 1 (strict Pydantic models) combine to create a system where every inter-agent handoff is typed, validated, and inspectable. The round-based directory structure extends this pattern for the iterative loop — it preserves the full evidence history, enables crash recovery at the round level, and makes the research progression visible in the filesystem. A developer or founder can `ls .founderflow/runs/<id>/rounds/` to see how many rounds ran and which agents participated in each, then `cat` any output file to see exactly what that agent found. This transparency is impossible with in-memory agent communication.

- **Validation Gates with Configurable Thresholds**
  - **What:** Before the Evidence Integrator runs in Round 1, a pre-synthesis gate checks that all three agent outputs meet minimum quality thresholds: each output must have the required fields populated, confidence scores must exceed a configurable minimum, and no agent should have timed out or returned an error. The gate prevents the Integrator from working with incomplete or garbage data. In subsequent rounds, the gate only checks the agents that re-ran.
  - **How:** `gates/validation.py` implements pre-synthesis checks inspired by the Factory's Pattern 7 (non-overridable precheck gates). Checks include: schema completeness (all required Pydantic fields populated), minimum content thresholds (e.g., competitor analysis must identify at least 2 competitors or alternatives), and agent health (no timeouts or errors in the event log for the current round). Gate results are logged as `gate.passed` or `gate.failed` events with the round number. Failed gates trigger a retry of the failed agent(s) before falling through to a degraded synthesis path that notes which perspectives are missing.
  - **Why:** The Factory's precheck gates (Pattern 7) prevent wasting compute on doomed operations. For FounderFlow, running the Evidence Integrator on incomplete data would produce misleading directives or validation briefs — the gate ensures the Integrator always has substantive input, which is especially important in the iterative loop where a bad directive in Round 1 would cascade into wasted re-investigation rounds.

## Architecture
- **Language/Runtime**: Python 3.11+ — required for `tomllib` built-in, `asyncio.TaskGroup`, and modern type syntax. The Factory reference implementation validates Python as the correct choice for multi-agent CLI tools.
- **Framework**: No agent framework (LangGraph/CrewAI) — the orchestration is a `while` loop with conditional `asyncio.gather()` over Claude Code subprocesses (`claude -p`). Research shows LangGraph adds value for complex graph-based workflows with conditional routing, but FounderFlow's iterative loop is a straightforward directive-driven re-invocation pattern that a Python `while` loop expresses more clearly than a graph definition. The Factory itself uses no agent framework.
- **Data Storage**: File-based only — `.founderflow/runs/<id>/` directories with round-specific subdirectories containing JSON files, HTML briefs, JSONL event logs, and TSV run summaries. No database for v1. Research confirms the Factory operates at production scale with flat files + filelock.
- **Key Libraries**:
  - `pydantic>=2.0` — strict models for all inter-agent data and research directives (Pattern 1)
  - `structlog` — JSON-structured logging (Pattern 9)
  - `filelock` — concurrent write protection (Pattern 8)
  - `typer[all]` — CLI framework with Rich integration (Pattern 10)
  - `rich` — live progress displays with round-aware layouts, formatted output panels, terminal brief rendering
  - `jinja2` — HTML brief template rendering from `templates/brief.html.j2`
  - `tomllib` (stdlib) — configuration parsing
  - `asyncio` (stdlib) — parallel agent execution with conditional re-invocation
  - `webbrowser` (stdlib) — auto-open HTML brief in default browser
  - `pytest` + `pytest-asyncio` — async test support

## Data Models

The following Pydantic v2 models (all with `ConfigDict(strict=True, extra="forbid")`) are defined in `models.py`:

- **`IdeaValidation`** — Idea Validator agent output: problem severity, target customer clarity, value proposition strength, risk factors, core assumptions, confidence score
- **`CompetitorAnalysis`** — Competitor Agent output: direct competitors, indirect competitors, substitutes, workarounds, pricing patterns, positioning gaps, confidence score
- **`CustomerDiscovery`** — Customer Discovery Agent output: interview targets, discovery questions, demand signals, weak-signal warnings, validation experiment design, confidence score
- **`FollowUpDirective`** — A single follow-up instruction: `agent_role` (enum: `idea_validator` | `competitor_analyst` | `customer_discovery`), `follow_up_question` (str), `priority` (enum: `high` | `medium`)
- **`ResearchDirective`** — Evidence Integrator's per-round output: `verdict` (enum: `needs_deeper` | `evidence_sufficient`), `directives` (list of `FollowUpDirective` — only agents listed here re-run), `resolved_contradictions` (list of str — what was clarified this round), `remaining_gaps` (list of str — what still needs work), `round_confidence` (int, 0-100)
- **`RoundResult`** — Captures a complete round: `round_num` (int), `active_agents` (list of agent role enums), `agent_outputs` (dict mapping role to output model), `directive` (ResearchDirective), `usage` (AgentUsage aggregate), `duration_ms` (int), `started_at` (datetime), `completed_at` (datetime)
- **`ValidationThesis`** — Final synthesis: thesis statement, confidence score 0-100, verdict enum (go/deeper/pivot/kill), contradictions list, unresolved assumptions list, synthesized risk assessment, research journey summary (list of `RoundResult` references), total rounds executed, convergence reason (enum: `evidence_sufficient` | `max_rounds` | `budget_exhausted`)
- **`StartupBrief`** — Complete brief artifact: `ValidationThesis` + all final agent outputs + all `RoundResult` objects + 7-day action plan + cost summary
- **`AgentUsage`** — Per-invocation cost tracking: `input_tokens`, `output_tokens`, `cache_read_tokens`, `cache_creation_tokens`, `total_cost_usd`, `duration_ms`, `num_turns`, `model`

## User Interface
The primary interface is a CLI. The founder's workflow:

1. **Submit idea**: `founderflow validate "A platform that connects freelance chefs with people who want home-cooked meals for dinner parties"`
2. **Watch Round 1 (broad)**: Rich live display shows "Round 1/3 (broad)" with three agents running in parallel — status indicators (spinner/checkmark/cross), elapsed time, and token counts per agent
3. **See Integrator analysis**: After all three agents complete, the Integrator runs and either declares evidence sufficient or produces a research directive identifying gaps
4. **Watch Round 2+ (directed)**: If gaps exist, the display shows "Round 2/3 (directed: idea_validator, customer_discovery)" — only the agents with identified gaps re-run, while the Competitor Agent shows as "idle (sufficient)" with a checkmark
5. **Convergence**: When the Integrator declares evidence sufficient (or max rounds reached), the loop ends and synthesis begins
6. **See verdict in terminal**: A color-coded verdict panel appears — green background for "go," yellow for "dig deeper," red for "pivot" or "kill" — with key risk indicators, the synthesized thesis, and a research journey summary showing rounds completed and convergence reason
7. **Read the full brief in the browser**: The HTML brief auto-opens. A polished, single-page report with clean typography, color-coded risk badges, structured sections, a collapsible research loop timeline, 7-day action plan, and per-round cost breakdown
8. **Share the brief**: Email the `.html` file to a co-founder, mentor, or advisor. They double-click to open it — no app install, no account, no server
9. **Review past runs**: `founderflow show <run-id>` re-opens the HTML brief in the browser; `founderflow show <run-id> --terminal` redisplays the terminal version; `founderflow show <run-id> --json` prints the raw JSON
10. **Check costs**: `founderflow costs` shows per-run and per-round cost breakdowns — useful for tuning `--max-rounds`
11. **List history**: `founderflow runs` lists all past validation runs with verdict badges (colored dots), round counts, and costs
12. **Control depth**: `founderflow validate "idea" --max-rounds 5` allows up to 5 rounds of investigation (default: 3)
13. **Customize agents**: Drop a modified prompt at `.founderflow/agents/idea_validator.md` to override the default Idea Validator behavior
14. **Suppress auto-open**: `founderflow validate "idea" --no-open` runs the pipeline without opening the browser (for headless/CI use)

## Non-Goals (v1)
- **Legal setup, incorporation, or entity formation** — FounderFlow validates ideas, it does not form companies
- **Fundraising, pitch decks, or investor materials** — out of scope for validation phase
- **MVP code generation or technical architecture** — the Builder agent is a future phase
- **Branding, landing pages, or marketing assets** — deferred to post-validation phases
- **Go-to-market execution** — the 7-day plan recommends actions, it doesn't execute them
- **Web UI or API server** — CLI-only for v1; FastAPI dashboard is a future phase
- **Multi-tenant or team features** — single-user local tool for v1
- **Real-time market data or live competitor monitoring** — agents use their training knowledge, not live web scraping
- **PDF export** — the self-contained HTML brief serves the same sharing purpose without requiring PDF rendering dependencies
- **Cross-run learning** — v1 does not use results from prior validation runs to inform new ones; each run is independent

## Open Questions
- **Anthropic API key**: Users need a Claude API key or Claude Code subscription. Should the CLI guide users through setup on first run, or assume Claude Code is already configured?
- **Model selection**: Should the default configuration use the same Claude model for all agents across all rounds, or should Round 1 agents use a faster/cheaper model while the Evidence Integrator and Round 2+ targeted agents use the most capable model? The Runner protocol supports per-agent and per-round model configuration, but what should the default be?
- **Token budget cap**: Should the `--max-rounds` flag be complemented by a `--max-cost` flag that terminates the loop when cumulative cost exceeds a dollar threshold? The event log tracks per-round costs, so this is implementable — the question is whether users want cost-based termination in addition to round-based termination.

---

*Research grounding: This specification draws on the Factory reference implementation's 14 production patterns (strict Pydantic v2, append-only event log, two-tier prompt resolution, Runner protocol, state machine, filelock persistence, structlog, Typer CLI, file-based agent communication, tiered history, composable prompts, validation gates, cost tracking, and introspection). The iterative research loop is the core architectural evolution from the prior draft — it extends the fan-out/fan-in pattern with a directive-driven re-invocation loop that mirrors the academic peer review process (initial review → gap identification → targeted follow-up → convergence). Research confirms wall-clock time for parallel agents equals the slowest agent, not the sum, and the iterative loop preserves this property: each round only re-invokes agents with gaps, so later rounds are faster and cheaper than Round 1. The decision to avoid LangGraph/CrewAI remains grounded in the Factory's own architecture — the loop is a simple `while` + conditional `asyncio.gather()`, which is more transparent and debuggable than a graph definition for this fixed topology. Competitor analysis (IdeaProof's 120-second single-pass, FounderSpace's multi-dimensional research) validates that iterative multi-perspective synthesis with cross-agent critique and directed deepening is the differentiating capability no existing tool provides.*

## Changes from Prior Draft
- **Iterative research loop**: Renamed "Parallel Three-Agent Validation Pipeline" to "Iterative Multi-Agent Research Loop." The pipeline is no longer a single linear pass — it's a convergent loop where Round 1 is broad/shallow and subsequent rounds are narrow/deep, driven by the Evidence Integrator's `ResearchDirective`.
- **Evidence Integrator → Research Advisor**: The Integrator now acts as a research advisor, not just a critic. It produces a structured `ResearchDirective` after each round that routes specific follow-up questions to specific agents, rather than just flagging contradictions in a synthesis report.
- **New Pydantic models**: Added `FollowUpDirective`, `ResearchDirective`, and `RoundResult` models. `ValidationThesis` now includes convergence reason and total rounds. `StartupBrief` includes all `RoundResult` objects for the full research journey.
- **DEEPENING state**: The state machine now includes a `DEEPENING` state between `VALIDATING` and `SYNTHESIZING` to explicitly track Rounds 2+ of directed re-investigation.
- **Round-based directory structure**: Agent outputs are now stored under `.founderflow/runs/<id>/rounds/<round_num>/` instead of flat in the run directory, preserving the full evidence history across rounds.
- **`--max-rounds` CLI flag**: Added `--max-rounds N` (default: 3) to control the iteration cap. The loop terminates when the Integrator declares convergence, the cap is reached, or the token budget is exhausted.
- **Round-level events**: Event log now includes `round.started`, `round.completed`, and `round.convergence_check` event types with round number, active agents, directive verdict, and per-round cost aggregation.
- **Per-round cost tracking**: `sum_round_costs()` function added. `runs.tsv` now includes `num_rounds` and `converged` columns. The HTML brief includes a per-round cost breakdown.
- **Additive prompt construction**: Agent re-invocations in Round 2+ receive their prior output plus the Integrator's follow-up question, building on prior work rather than starting fresh. A new `prior_work.md` prompt section template handles this injection.
- **Round-aware Rich display**: CLI progress display now shows current round number, which agents are active vs. idle, and per-round cost accumulation.
- **Removed "Iterative refinement loops" from Non-Goals**: This is now a core feature, not a future item. Replaced with "Cross-run learning" as a v1 non-goal.
- **Added `--max-cost` Open Question**: Whether to complement round-based termination with cost-based termination.
